# v0.2 — 루프 엔진 + define 실험 + 관측 인프라

## 목표

1. 범용 루프 엔진(loop.py + judge.py)을 만든다
2. experiments/define-prd/에서 "PRD 생성 스크립트"를 루프로 최적화
3. 로그를 Supabase에 쌓고 Vercel 대시보드에서 확인
4. 검증된 스크립트를 plugin에 편입

## 핵심 구조

### auto-research 매핑

| auto-research | moomoo-ax | 누가 관리 |
|---------------|----------|----------|
| program.md | `program.md` — 오너 규칙 (톤, 구조, 제약) | 오너 |
| train.py | `script.md` — 생성 스크립트 (AI가 루프에서 개선) | AI |
| prepare.py | `rubric.yml` — 평가 체크리스트 | 오너 (루프 안 불변) |
| results.tsv | `logs/` + Supabase | 자동 |

### 루프 흐름

```
script.md로 산출물 생성 → rubric으로 산출물 eval → 점수
  → 점수 > best → keep (script.md 저장 + best 갱신)
  → 점수 ≤ best → discard (실패 항목 → script.md 수정 피드백)
→ 종료: 점수 ≥ 임계값 or max iteration
```

루프가 개선하는 건 "이번 산출물"이 아니라 **"산출물을 잘 만드는 스크립트"**.

```

### 실험 → 프로덕션 흐름

```
① experiments/define-prd/ 생성 (program.md + script.md 초안 + rubric.yml)

## 첫 실험: define-prd

### program.md (오너 규칙)

```markdown
# PRD 생성 규칙

## 구조
PRD는 다음 섹션을 포함해야 한다:
- 문제 정의 (as-is → to-be)
- 타겟 유저 (페르소나 + JTBD)
- 핵심 기능 (우선순위 포함)
- MVP 스코프 + 스코프 아웃
- 성공 지표 (측정 가능)
- 기술 제약

## 톤
- 간결하고 실행 가능한 문장
- 불필요한 서론/배경 최소화
- 각 항목에 "왜"가 1줄이라도 있을 것

## 제약
- 마크다운 형식
- 2000자 이내
```

### rubric.yml

```yaml
items:
  - question: "해결하려는 문제가 한 문장으로 명확히 정의되어 있는가?"
    critical: true
  - question: "as-is / to-be가 구분되어 있는가?"
    critical: false
  - question: "타겟 유저가 구체적 페르소나로 정의되어 있는가?"
    critical: true
  - question: "유저의 핵심 목표(JTBD)가 명시되어 있는가?"
    critical: true
  - question: "핵심 기능이 우선순위와 함께 나열되어 있는가?"
    critical: true
  - question: "MVP 스코프가 명확히 경계지어져 있는가?"
    critical: true
  - question: "스코프 아웃 항목이 명시되어 있는가?"
    critical: false
  - question: "성공 지표가 측정 가능한 형태인가?"
    critical: true
  - question: "기술 스택 또는 제약이 명시되어 있는가?"
    critical: false
  - question: "데이터 모델 또는 주요 엔티티가 정의되어 있는가?"
    critical: false
  - question: "2000자 이내인가?"
    critical: false
  - question: "문서 구조가 일관되고 읽기 쉬운가?"
    critical: false
```

### script.md (초안 — AI가 개선할 대상)

```markdown
# PRD 생성 스크립트

CPS를 입력받아 PRD 문서를 생성한다.

## 절차
1. CPS에서 Context, Problem, Solution을 추출
2. Problem → 문제 정의 (as-is/to-be)
3. Context → 타겟 유저 페르소나 + JTBD 도출
4. Solution → 핵심 기능 목록 + 우선순위
5. 기능 목록에서 MVP 스코프 경계
6. 성공 지표 도출
7. 기술 제약 명시

## 출력 형식
마크다운. 2000자 이내.
```

### input/cps.md

대시보드 프로젝트의 CPS. (실행 시 작성)
