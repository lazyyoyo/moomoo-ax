## 핵심 구조

### auto-research 매핑

| auto-research | moomoo-ax | 누가 관리 |
|---------------|----------|----------|
| program.md | `program.md` — 오너 규칙 (톤, 구조, 제약) | 오너 |
| train.py | `script.md` — 생성 스크립트 (AI가 루프에서 개선) | AI |
| prepare.py | `rubric.yml` — 평가 체크리스트 | 오너 (루프 안 불변) |
| results.tsv | `logs/` + Supabase | 자동 |

### 루프 흐름

```
script.md로 산출물 생성 → rubric으로 산출물 eval → 점수
  → 점수 > best → keep (script.md 저장 + best 갱신)
  → 점수 ≤ best → discard (실패 항목 → script.md 수정 피드백)
→ 종료: 점수 ≥ 임계값 or max iteration
```

루프가 개선하는 건 "이번 산출물"이 아니라 **"산출물을 잘 만드는 스크립트"**.

### 디렉토리 구조

```
moomoo-ax/
├── loop.py              ← 범용 오케스트레이터
├── judge.py             ← 루브릭 → LLM Judge → 점수
├── db.py                ← Supabase 로그 래퍼
├── labs/                ← 실험 (루프 돌리는 곳)
│   └── define-prd/
│       ├── program.md   ←   오너 규칙 (불변)
│       ├── script.md    ←   AI가 개선하는 스크립트
│       ├── rubric.yml   ←   평가 기준 (불변)
│       ├── input/       ←   입력 파일 (CPS 등)
│       ├── best/        ←   최적 산출물 + script
│       └── logs/        ←   iteration 로그 (로컬)
├── plugin/              ← 프로덕션 (검증된 스킬)
├── dashboard/           ← Vercel 대시보드
└── versions/            ← 버전별 plan + 기록
```

## 첫 실험: define-prd

### program.md (오너 규칙)

```markdown
# PRD 생성 규칙

## 구조
PRD는 다음 섹션을 포함해야 한다:
- 문제 정의 (as-is → to-be)
- 타겟 유저 (페르소나 + JTBD)
- 핵심 기능 (우선순위 포함)
- MVP 스코프 + 스코프 아웃
- 성공 지표 (측정 가능)
- 기술 제약

## 톤
- 간결하고 실행 가능한 문장
- 불필요한 서론/배경 최소화
- 각 항목에 "왜"가 1줄이라도 있을 것

## 제약
- 마크다운 형식
- 2000자 이내
```

### script.md (초안)

```markdown
# PRD 생성 스크립트

CPS를 입력받아 PRD 문서를 생성한다.

## 절차
1. CPS에서 Context, Problem, Solution을 추출
2. Problem → 문제 정의 (as-is/to-be)
3. Context → 타겟 유저 페르소나 + JTBD 도출
4. Solution → 핵심 기능 목록 + 우선순위
5. 기능 목록에서 MVP 스코프 경계
6. 성공 지표 도출
7. 기술 제약 명시

## 출력 형식
마크다운. 2000자 이내.
```

대시보드 프로젝트의 CPS. (실행 시 작성)
