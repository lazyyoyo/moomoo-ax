# v0.2 — 루프 엔진 + define 루프 검증 + 관측 인프라

## 목표

1. 범용 루프 엔진(loop.py + judge.py)을 만든다
2. define(PRD 작성) 태스크로 "루프가 수렴하는지" end-to-end 검증
3. 로그를 Supabase에 쌓고 Vercel 대시보드에서 확인

## 단계별 output + eval 정의 (v0.2~v0.4 전체 그림)

| 단계 | input | output | eval 방식 | eval 기준 |
|------|-------|--------|----------|----------|
| **define** | 러프 아이디어/CPS | PRD 문서 | LLM Judge (루브릭) | 문제 명확성, 유저 정의, 성공 지표, 기술 제약, 스코프 |
| **design** | PRD | 디자인 명세 + 컴포넌트 코드 | LLM Judge + 시각 diff | 레이아웃, 타이포, 색상, 상태 처리, 접근성, 반응형 |
| **implement** | 디자인 명세 | 코드 | 정적 게이트 + LLM Judge | lint, typecheck, build, 기능 완성도, 스펙 준수 |

v0.2에서는 **define만**. 나머지는 v0.3, v0.4에서.

## 1. 루프 엔진

### 구조

```
input → LLM 생성 → eval(루브릭 → 점수) → 판정
  → 점수 > best → keep
  → 점수 ≤ best → discard (실패 항목 → 다음 반복 피드백)
→ 종료: 점수 ≥ 임계값 or max iteration
```

### 파일 구조

```
moomoo-ax/
├── loop.py              ← 오케스트레이터 (범용)
├── judge.py             ← 루브릭 → LLM Judge → 점수
├── db.py                ← Supabase 로그 저장
├── tasks/
│   └── {task-name}/
│       ├── program.md   ← 역할 + 규칙
│       ├── rubric.yml   ← eval 체크리스트
│       └── input/       ← 입력 파일
├── output/              ← 산출물 (task별 하위)
├── dashboard/           ← Vercel 대시보드 앱
└── versions/
```

### loop.py (의사코드)

```python
def run(task_dir, project, user, max_iter=10, threshold=0.85):
    program = read(task_dir / "program.md")
    rubric = read(task_dir / "rubric.yml")
    input_ctx = read(task_dir / "input/")

    best_score = 0
    feedback = None

    for i in range(max_iter):
        output = call_claude(program, input_ctx, feedback)
        score, failed = judge(rubric, output)

        verdict = "keep" if score > best_score else "discard"
        if verdict == "keep":
            best_score = score
            save_output(task_dir, output)

        # 로그: 로컬 JSON + Supabase
        log = {iteration, score, verdict, failed, tokens, duration}
        save_local(task_dir / f"logs/{i:03d}.json", log)
        db.insert("iterations", {**log, project, user, task})

        if score >= threshold:
            break

        feedback = failed

    return best_score
```

### judge.py

루브릭 항목마다 LLM에게 Yes/No 판정. 점수 = Yes 비율. critical No → 강제 실패.

### rubric.yml 형식

```yaml
items:
  - question: "문제 정의가 명확한가?"
    critical: true
  - question: "타겟 유저가 구체적으로 정의되어 있는가?"
    critical: true
  - question: "성공 지표가 측정 가능한가?"
    critical: false
```

## 2. Supabase 스키마

### 테이블: `iterations`

ax-loop 매 반복 로그. 모든 프로젝트/유저/태스크가 여기에 쌓임.

| 컬럼 | 타입 | 설명 |
|------|------|------|
| id | uuid (PK) | 자동 생성 |
| created_at | timestamptz | |
| project | text | 프로젝트명 (dashboard, rubato 등) |
| user | text | 실행자 (yoyo, husband) |
| task | text | 태스크명 (dashboard-prd 등) |
| stage | text | define / design / implement |
| iteration | int | 반복 번호 |
| score | float | 0.0 ~ 1.0 |
| verdict | text | keep / discard |
| failed_items | jsonb | 실패한 루브릭 항목 배열 |
| tokens_input | int | 입력 토큰 |
| tokens_output | int | 출력 토큰 |
| duration_sec | float | 소요 시간 |
| model | text | claude-sonnet-4 등 |

## 3. 대시보드 (최소)

Vercel에 배포. Supabase에서 읽기만.

### 뷰

1. **전체 요약** — 프로젝트별 최근 run, 총 토큰, 평균 점수
2. **점수 추이** — 특정 run의 iteration별 점수 차트 (수렴 확인용)
3. **프로젝트별** — 프로젝트 선택 → 해당 run 목록
4. **유저별** — yoyo / 남편 분리

### 기술

- Next.js (App Router)
- Supabase JS client (read-only)
- 차트: recharts 또는 lightweight 라이브러리
- Vercel 배포
