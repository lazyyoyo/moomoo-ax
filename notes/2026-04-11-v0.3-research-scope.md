# v0.3 리서치 스코프

2026-04-11 작성. v0.2 E/F 마감 직후, v0.3 착수 **전**.

## 왜 리서치가 먼저인가

v0.2 가 발견한 두 구조 결함 (`notes/2026-04-11-v0.2-e-f-codification-insight.md`):

1. 자연어 압축 ≠ codification
2. `claude -p` one-shot ≠ Claude Code tool loop

**v0.3 재설계의 모양이 Claude Code CLI 의 실 기능/제약에 달려 있음**. 가설을 먼저 검증하지 않고 설계 들어가면 v0.2 와 같은 한계에 또 부딪힐 위험.

## 리서치의 3 축

### 축 1 — Claude Code CLI 의 tool-enabled 호출 가능성

**핵심 질문**: Python subprocess 가 Claude Code CLI 를 "tool-enabled 모드" 로 호출해서 skill 을 진짜로 실행시킬 수 있는가?

**세부 질문**:
- `claude -p` 에 `--allowedTools Read,Write,Bash,Agent` 같은 flag 가 동작하는가?
- `--permission-mode {auto, acceptEdits, plan, bypassPermissions}` 는 subprocess 에서 어떻게 평가되는가?
- skill 을 강제 load 하는 방법이 있는가? (`--skill team-ax:ax-implement` 같은)
- 특정 plugin 을 활성화/비활성화하는 flag 는?
- `--output-format json` 에 tool call 이력 / cost / token 이 포함되는가?
- `--resume {session_id}` 로 세션 이어붙이기 가능한가? 그게 loop 에 도움 되는가?
- Claude Code 안에서 또 `claude` subprocess 호출 시 **재귀 문제** 없는가? (session id 충돌, permission 상속 등)
- stdin 으로 대량 입력 (수 KB ~ 수십 KB fixture) 주입 제한은?
- stdout JSON 출력의 max size 제한은?

**산출물**: `docs/claude-code-spec/cli.md` + `docs/claude-code-spec/permissions.md` + 실 테스트 기록

### 축 2 — Claude Code 의 skill / plugin / hook / command / agent 구조

**핵심 질문**: SKILL.md / plugin.json / hook / command / agent 의 **공식 규약** 은 무엇이며, levelup loop 가 어디까지 모사/활용해야 하는가?

**세부 질문**:
- SKILL.md frontmatter 의 모든 필드 (name, description, allowed-tools 등) 와 semantics
- SKILL.md 본문의 규약 — `/<cmd>`, `<tool>`, `$ARGUMENTS` 같은 placeholder
- `references/*` 파일의 로드 방식 (자동 주입? Claude 가 Read tool 로만?)
- plugin.json 스키마 (name, version, skills, commands, hooks, agents, mcp_servers)
- `.claude-plugin/` 디렉토리 레이아웃
- hook 이벤트 종류 (PreToolUse, PostToolUse, UserPromptSubmit, Stop 등) 와 실행 컨텍스트
- slash command 작성법, frontmatter, 인자 바인딩
- subagent markdown 규약 (agents/), tool whitelist, description 기반 선택
- MCP server 플러그인 내장 방식
- Claude Code 마이너 버전 간 breaking change 추적 방법

**산출물**: `docs/claude-code-spec/{skill,plugin,hook,command,agent,mcp}.md`

### 축 3 — 하이브리드 loop 가능성 검증

**핵심 질문**: Python 하네스 + Claude Code skill 실 실행 하이브리드가 **실제로 돌아가는가**?

**실험 단계**:
1. **최소 echo 실험** — Python 이 `claude --allowedTools Read -p "Read this file and echo it"` 호출 → Claude 가 실제로 tool call 수행했는지 JSON 으로 확인
2. **stateful 실험** — Python 이 Claude 에 "파일 하나 만들고 내용 적어봐" 지시 → 파일 실제 생성 여부 확인
3. **재귀 실험** — 현재 Claude Code 세션 안에서 subprocess 로 또 `claude` 호출 → 재귀 depth 제약 확인
4. **skill load 실험** — `claude --skill some:skill -p "..."` 로 특정 skill 강제 load 시도 → 가능 여부 확인
5. **비용 집계 실험** — 여러 tool call 이 있는 세션의 비용이 output JSON 에 정확히 반환되는지
6. **MCP 실험** — Playwright MCP 를 띄운 상태에서 `claude` subprocess 가 해당 MCP 를 사용할 수 있는지 (현 세션 MCP 상속 vs 격리)

**산출물**: `notes/v0.3-experiments/*.md` 로그 + `notes/2026-04-12-v0.3-feasibility.md` 판정 리포트

### 3 축의 의존 관계

```
축 2 (spec 레퍼런스)  ──┐
                        ├──→  축 3 (하이브리드 실험)  ──→  v0.3 go/no-go 판정
축 1 (CLI 가능성)    ──┘
```

축 2 가 "이론", 축 1 이 "CLI 실 기능", 축 3 가 "조합 검증". 축 3 의 결과에 따라 v0.3 설계가 둘 중 하나로 갈림:

- **가능 (Path A)**: 하이브리드 loop.py 재설계 — Python 하네스 유지 + 각 iter 는 Claude Code tool loop 으로 실 실행
- **불가능 (Path B)**: 완전 skill 기반 loop — Python 은 얇은 entry point, 본체는 Claude Code 세션 안에서
