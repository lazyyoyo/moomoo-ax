# 루프 정의 — auto-research → moomoo-ax

2026-04-09

## auto-research 원본 구조

### 구성 요소

| 요소 | 역할 | 수정 가능? |
|------|------|-----------|
| **program.md** | 에이전트 행동 규칙. "너는 연구자야. train.py만 수정해. 5분 안에 끝나는 실험만 해." | 사람이 천천히 개선 |
| **train.py** | 유일한 수정 대상. 에이전트가 매 반복 수정. | 에이전트가 매 반복 |
| **prepare.py** | 평가 함수 + 고정 상수. 데이터 로드, 학습, 점수 산출. | 수정 불가 (하네스) |
| **results.tsv** | 실험 로그. 매 반복 자동 기록. | 자동 기록 |

### 루프

```
수정(train.py) → git commit → 실행(5분) → 평가(숫자 1개) → 판정
  → 점수 개선 = keep (커밋 유지)
  → 점수 동일/하락 = discard (git reset)
  → 빌드 실패 = crash (git reset + 에러 로그)
→ 다음 반복 (NEVER STOP)
```

### 왜 작동하는가

1. **eval이 숫자 1개** — gradient가 있음. "좋아졌다/나빠졌다"가 명확
2. **수정 범위가 좁음** — train.py 1개 파일만. diff 검토 쉬움
3. **실행이 빠름** — 5분. 빠른 반복 = 많은 시도
4. **하네스(prepare.py)가 불변** — 평가 기준이 흔들리지 않음
5. **자동 판정** — 사람 개입 0. keep/discard가 기계적

## moomoo-ax에서의 문제

v0.1에서 이걸 "제품 개발 전체"에 매핑하려다 실패. 근본 차이:

| auto-research | v0.1 시도 |
|---------------|----------|
| eval = loss (연속값) | eval = lint pass (이진) |
| 수정 = train.py 1개 | 수정 = 프로젝트 전체 |
| 실행 = 5분 | 실행 = 수분~수십분 |
| 판정 = 자동 | 판정 = 첫 통과시 즉시 종료 (루프가 아님) |

## 핵심 깨달음

auto-research가 작동하는 이유는 **"좁은 수정 범위 + 숫자 eval + 빠른 반복"** 이 세 가지가 동시에 성립하기 때문.

제품 개발 전체에 한 번에 적용하면 세 가지 다 깨짐:
- 수정 범위가 너무 넓음
- eval이 이진이거나 주관적
- 반복이 느림

## 해결: 단일 업무 단위로 쪼개기

"제품 개발"이 아니라 **"PRD 작성"** 같은 단일 업무에 루프를 적용하면:

| | PRD 작성 루프 |
|---|---|
| program.md | "너는 PRD 작성자야. CPS를 입력받아 PRD를 출력해." |
| train.py (수정 대상) | PRD 문서 1개 |
| prepare.py (eval) | PRD 루브릭 체크리스트 (항목별 Yes/No → 합산 점수) |
| 판정 | 점수 > 이전 best → keep, 아니면 discard |

이러면 auto-research의 세 조건이 다시 성립:
- ✅ 수정 범위 좁음 (PRD 1개 문서)
- ✅ eval이 숫자 (체크리스트 합산 점수)
- ✅ 반복 빠름 (문서 생성은 수십초)

## 다른 단일 업무 예시

| 업무 | 수정 대상 | eval | 예상 반복 시간 |
|------|----------|------|---------------|
| PRD 작성 | PRD.md | 루브릭 점수 | ~30초 |
| 디자인 | 컴포넌트 코드 | 시각 diff + 체크리스트 | ~1분 |
| 구현 | 기능 코드 | lint + typecheck + test | ~1분 |
| 문서 갱신 | docs/*.md | 완성도 체크리스트 | ~20초 |

각 업무마다 **별도의 program.md + eval**을 정의. 루프 구조(오케스트레이터)는 공유.

## 다음 단계

1. **루프 구조 만들기** — 오케스트레이터가 "수정 → eval → keep/discard → 반복"을 범용으로 실행
2. **첫 번째 업무 정의** — 대시보드 프로젝트의 특정 단계 (예: PRD 작성 또는 디자인)
3. **돌려보고 수렴 확인** — 점수가 올라가는가?
4. **관측 인프라** — 로그 → Supabase → 대시보드
