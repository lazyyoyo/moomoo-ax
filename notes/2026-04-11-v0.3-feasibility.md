## 최종 판정: Path A (하이브리드)

```
Python 하네스 (src/loop.py)
    │
    ├─ rubric 평가, 루프 제어, DB 로깅, diff 수집 (변경 최소)
    │
    └─ subprocess.run([
          "claude", "-p", f"/team-ax:ax-implement {args}",
          "--plugin-dir", "plugin",
          "--allowedTools", "Read", "Write", "Edit", "Bash",
                            "Grep", "Glob", "Agent",
                            "mcp__plugin_playwright_playwright__*",  # stage 별
          "--permission-mode", "acceptEdits",
          "--output-format", "stream-json", "--verbose",
          "--bare",  # 재현성
          "--setting-sources", "project,local",
       ])
         │
         └─ 자식 claude-p 세션
              - team-ax 플러그인 자동 로드 (plugin-dir)
              - /team-ax:ax-implement 슬래시 호출로 스킬 즉시 실행
              - SKILL.md 본문 내 scripts/*.py 호출 → deterministic
              - MCP tool 필요 시 실 호출 (Playwright 등)
              - stream-json 으로 per-tool 감사 데이터 반환
```

## 결정 근거 6가지

### 1. 전면 재설계보다 작은 변경이 더 유리하다

v0.1~v0.2 의 `src/loop.py`, `src/judge.py`, `src/db.py` 골격은 검증됨 (v0.2 E 에서 1 cycle 완주). 재설계 범위는 대략:

- `src/claude.py:38` — 플래그 3~4개 추가 (한 곳, 한 줄)
- `src/loop.py` — stream-json 파싱 + `--plugin-dir` 경로 주입 (50줄 내외)
- `plugin/skills/ax-implement/SKILL.md` — 자연어 HARD RULES 를 `scripts/*.py` 호출로 교체 (본 작업)
- `plugin/skills/ax-implement/scripts/check_*.py` — 새로 작성 (rubric axis 당 1 스크립트)

Path B (전면 skill 화) 는 `src/loop.py` 를 버리고 Claude Code 세션 안에서 loop 을 돌려야 하는데 — 이건 Python 의 DB 로깅, 오케스트레이션, 재현성을 모두 재구현해야 한다. **불필요한 재작성 비용**.

### 2. 비용 구조가 감당 가능

- iter 당 baseline: **$0.3 ~ $0.6**
  - 일반 stage (ax-implement, ax-design): $0.3 ~ $0.4
  - MCP heavy stage (ax-qa): $0.5 ~ $0.6
- 기본 max-iter 3 + threshold 0.95 이면 run 당: **$0.9 ~ $1.8**
- rubato 한 기능당 run 비용으로 수용 가능

**단점**: 호출마다 38K cache creation 이 baseline 으로 박힘. 자식 세션은 부모와 cache share 가 안 돼서 반복 호출 시 cache read 가 누적되지 않는다. 해결책:
- `--exclude-dynamic-system-prompt-sections` 플래그 → prompt cache reuse 향상 (실험 필요)
- 같은 fixture 를 여러 iter 에 걸쳐 사용할 때 cache hit 율이 올라갈지는 미확인

### 3. Progressive Codification 이 본 작업이다

exp-04 Test A 에서 본 자연어 편차 (SKILL.md 지시 없이 코드 펜스 추가) 는 Path A/B 선택과 무관한 구조 문제. **v0.3 의 진짜 본 작업**은 SKILL.md 본문에서 deterministic 부분을 스크립트로 뽑는 것:

```markdown
<!-- 전 -->
## HARD RULES
- R-LEN: output must be under 100 lines
- R-DRY: no duplicate imports

<!-- 후 -->
## HARD RULES
Run these checks before finalizing:
!`python scripts/check_r_len.py $CLAUDE_SKILL_DIR/tmp/output.py`
!`python scripts/check_r_dry.py $CLAUDE_SKILL_DIR/tmp/output.py`
If either exits non-zero, fix and re-run until both pass.
```

`allowed-tools: Bash(scripts/*.py)` 로 permission 보장 → Claude 는 "해석" 할 여지 없이 스크립트 실행만 함. 자연어 rule 의 비일관성 원천 제거.

**이 리팩터는 Path A/B 중 어느 쪽을 고르더라도 해야 한다**. Path A 는 이 리팩터에 더 친화적 — Python 하네스가 scripts/ 디렉토리를 직접 테스트/수정할 수 있다.

### 4. Plugin-dir 은 /reload-plugins 없이도 충분

매 iter 마다 새 자식 세션을 띄우므로 플러그인 수정이 즉시 반영된다. `/reload-plugins` 는 장기 세션 전용. 이게 levelup loop 와 자연스럽게 맞는다.

### 5. 재현성이 확보 가능하다

자식 세션이 부모 환경을 자동 복제하는 것은 양날의 검 (exp-05 발견). 하지만 다음 조합으로 재현성 확보:

```
--bare
--setting-sources project,local
--plugin-dir plugin
--permission-mode acceptEdits
--allowedTools <명시적 목록>
```

`--bare` 가 auto-memory / hooks / keychain / plugin sync / CLAUDE.md auto-discovery 를 스킵 → 실행 환경이 플래그로 완전히 정의됨. CI 친화적.

### 6. ax-qa 의 구조적 재설계가 불필요

v0.2 F 의 가장 큰 걱정 (`labs/ax-qa` 가 Playwright 못 씀) 이 exp-06 에서 해결됨. 같은 플래그 조합에 MCP tool 만 더하면 Playwright/axe/Lighthouse 사용 가능.

## 벽 재평가

### 벽 1 — 자연어 ≠ codification: **유효**

exp-04 Test A 의 자발적 코드 펜스 추가가 증거. 더 나아가 R-LEN 같은 규칙을 자연어로 적어두면 Claude 가 매번 "대략 100줄 정도면 되겠지" 식으로 느슨하게 해석. **Progressive Codification 은 v0.3 의 본 작업**.

### 벽 2 — Python one-shot ≠ tool loop: **반박됨**

`src/claude.py` 의 플래그 누락이 원인. Claude Code CLI 자체에는 제약 없음. **"플래그 3~4개 추가" 가 fix**.

## 리스크와 mitigation

| 리스크 | 확률 | 영향 | Mitigation |
|---|---|---|---|
| 자식 세션 cache reuse 가 낮아 iter 당 비용 overshoot | 중 | 중 | `--exclude-dynamic-system-prompt-sections` 실험. max-iter 를 2 로 낮춤. |
| Claude 가 스크립트 실행 없이 "결과를 추정" | 중 | 높 | `hooks.json` 의 PostToolUse 로 스크립트 실행 여부 강제 검증 |
| MCP tool 이름이 플러그인 버전마다 변경 | 낮 | 중 | team-ax 에 Playwright MCP 내장 vs user MCP 의존 결정 (v0.4 논의) |
| `--plugin-dir plugin` 으로 로드된 team-ax 가 개발 중 SKILL.md 수정 시 stale cache | 낮 | 낮 | 매 iter 새 subprocess 라 stale 불가능 |
| 병렬 iter 시 Playwright browser N 개 동시 → 리소스 고갈 | 낮 | 낮 | parallel cap = 2 |
| `--bare` 가 예상 외 기능을 끄는 경우 (skill invocation 포함?) | 낮 | 중 | exp 추가: `--bare --plugin-dir plugin -p "/team-ax:ax-feedback ..."` 동작 확인 |

## 범위 밖 (이번 리서치에서 의도적으로 안 다룸)

- Claude Code 의 session resume (`-r`) 으로 loop 연속성 구현 → 복잡하고 불필요. 매 iter 새 subprocess 가 간단.
- Claude Agent SDK 사용 → `claude-code-guide` 에이전트 조사에서 언급됐지만, Python subprocess 패턴이 이미 충분하므로 SDK 도입은 보류.
- team-product 의 다른 스테이지 (ax-design, ax-deploy) 포팅 → v0.4 스코프.
- 북극성 지표 (오너 개입 횟수) 자동 측정 방법 → v0.5 스코프.

## 이번 세션의 수확 한 줄

> v0.2 가 본 "두 개의 벽" 중 하나는 **코드 버그**였고, 다른 하나는 **codification 이 필요한 본 작업**이었다. Phase 0 하루 만에 이걸 분리할 수 있었던 게 가장 큰 수확. v0.3 는 "재설계" 가 아니라 **점진 리팩터 + deterministic 추출**이다.
