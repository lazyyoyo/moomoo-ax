# v0.1 회고 + 리셋 결정

2026-04-09

## v0.1에서 배운 것

### 실패 원인
- 구현 범위가 너무 커서 테스트 자체가 안 됨
- tmux 인프라부터 쌓다가 루프 1회 검증도 못 함
- 정적 게이트 통과 = 즉시 종료 → one-shot이지 루프가 아님
- eval에 gradient가 없음 (이진 pass/fail만)

### Karpathy 피드백 핵심
- "멍청한 베이스라인부터" — subprocess.run으로 1회 end-to-end 성공부터
- "gradient 없는 eval은 랜덤 서치" — error count라도 추적해야 수렴 가능
- "코드가 강제한다" 원칙은 좋다

## 리셋 방향

기존 구현 전부 클리어하고 처음부터.

### 순서

```
① 루프 정의 — auto-research 원본 구조를 명확히 하고, moomoo-ax 매핑
② 루프 구조 만들기 — 최소 동작하는 루프 (단일 업무 단위)
③ 뭘 만들지 정하기 — 첫 프로젝트 = 대시보드 (dogfooding)
④ 만들면서 강화하기 — 실제 적용 → 개선점 도출 → 루프 진화
```

### 핵심 변경

| 이전 (v0.1) | 이후 |
|-------------|------|
| 전체 파이프라인 (Phase 0→3) | 단일 업무 단위 루프 (PRD 작성, 디자인 등) |
| 정적 게이트만 (이진) | gradient 있는 eval (점수 기반) |
| tmux 기반 워커 | subprocess.run (베이스라인) |
| 로그 JSON 파일만 | → Supabase DB → Vercel 대시보드 |
| moomoo-ax 단독 | 나+남편 프로젝트에 적용하면서 진화 |

### 관측 인프라 계획

로그 → Supabase DB → Vercel 대시보드
- 프로젝트 진행 상황, 토큰 소비량, keep/discard 이력
- 나와 남편 프로젝트 모두 여기서 추적

### 디자인 단계 eval 기준

"어떤 프로세스를 타야 & 어떤 평가 기준을 가져가야 오너의 피드백을 최소화하면서 기대치에 도달할 수 있는가"
