## 배경

남편이 MAO 플러그인의 다음 단계로 **하네스 엔지니어링** 도입을 제안.
현재 team-product, team-design으로 제품 개발은 진행되지만, 디자인 완성도가 낮아 오너 시간을 과도하게 소비하는 문제가 있음.

## 핵심 페인포인트

- 디자인 디테일 완성도 부족 → 오너의 수동 리뷰 시간 과다
- Claude Max $220 × 2계정으로도 토큰 부족
- 단일 모델(Claude) 의존

## 목표

1. **하네스 엔지니어링**: 어떤 모델을 쓰든 멱등한 결과물이 나오도록 제약/규칙/평가 프레임워크 구축
2. **멀티모델 인프라**: Claude + Codex + Gemini 오케스트레이션
3. **auto-research 도입**: Karpathy의 autoresearch 패턴으로 자율 품질 개선 루프
4. **토큰 최적화**: 모델 티어링으로 비용 효율화

## 설계 철학

**"오너는 CPS(무엇을 만들지)만 던진다. 나머지는 전부 시스템이 한다."**

- 페르소나 도출 → CPS 컨텍스트에서 자동 생성 (사전 정의 X)
- 평가 기준 생성 → 페르소나 × 기능에서 자동 생성 (사전 정의 X)
- 구현 → 멀티모델 워커 (Claude/Codex/Gemini)
- 평가 → 자동 루프 (정적 게이트 → 시각 게이트 → 규칙 Judge → 페르소나 Judge)
- 재작업 → 자동 (fail 시 재작업 프롬프트 생성 → 루프)
- 오너 개입 = 최종 결과물 확인만
- 결과가 쓰레기면 → 프로세스(eval 로직/페르소나 생성)를 개선 (시스템 레벨)

## 결정 사항

- **새 플러그인으로 구현** (기존 team-product/team-design 리스크 회피)
- **첫 버전 스코프: 멀티모델 인프라 먼저** → 그 위에 하네스/auto-research 얹기
- **완전 자동화 목표** — 오너 승인 게이트 없음, 오너 개입 최소화
- **페르소나/평가기준 동적 생성** — 기능 컨텍스트에 따라 매번 자동 도출

## 평가 프레임워크 (2-Level)

### 레벨 1: AX 시스템 자체 평가 — "이 시스템이 팀을 돕고 있는가?"
자동 측정 가능한 지표만:
- 토큰 소비량 (모델별: Claude/Codex/Gemini) → ↓ 또는 동일 토큰 대비 품질 ↑
- 오너 개입 횟수 ("다시 해" / 되돌리기 횟수) → ↓
- eval 루프 내 자동 재작업 횟수 (린트/빌드/체크리스트 실패 → 재시도) → 추이 트래킹
- 첫 결과 수용율 (오너 수정 요청 없이 수용된 비율) → ↑
- 시스템 밖에서 로그/리포트로 트래킹

### 레벨 2: 제품 결과물 평가 — "이 결과물이 좋은가?"
- 시스템 안(플러그인 내장 eval)에서 자동 평가
- 처음부터 완벽한 eval 불필요 → 측정 가능한 것부터 시작, eval 자체를 진화
- 1단계: 린트 통과율, 빌드 성공, 스크린샷 diff, 상태 처리 존재 여부
- 2단계: 페르소나 체크리스트, 레퍼런스 비교, 브랜드 토큰 준수율 (결과 보며 추가)
- eval 함수의 진화 = 시스템의 진화

### NotebookLM 정리 핵심

**하네스 엔지니어링 4단계:**
1. CPS(Context-Problem-Solution) 프레임워크로 요구사항 세분화
2. 대원칙 수립 + 도메인 주도 설계(DDD) → AI가 딴길로 안 새게
3. 린터로 코드 통제 (파일명, import 순서, 구조 강제)
4. 조직 맞춤형 평가(Eval) 프레임워크 → 품질 트래킹

**auto-research 핵심 원리:**
- 평가 함수(Eval)를 바꾸면 최적화 방향이 바뀐다
- 사람은 올바른 목표 지표 설정 + 평가 구조 제공, AI가 루프 돌며 자체 개선
