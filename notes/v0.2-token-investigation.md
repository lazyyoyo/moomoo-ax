# v0.2 B — 토큰 집계 조사

**날짜**: 2026-04-11
**배경**: v0.1 첫 ax-qa run 에서 script 호출이 `input 5 / output 1230` 으로 기록됨. 프롬프트 규모가 fixture 전체(수천 줄)를 포함하는데 input 이 5 는 말이 안 됨. prompt caching 영향 추정.

## 실측

`claude -p "print hello" --output-format json` 의 raw `usage` 필드:

```json
{
  "input_tokens": 6,
  "cache_creation_input_tokens": 36796,
  "cache_read_input_tokens": 0,
  "output_tokens": 6,
  "total_cost_usd": 0.31009780000000003
}
```

**발견**:
- `input_tokens` 는 "이번 호출에서 **캐시 안 된** 입력" 만. Claude CLI 가 system prompt + 도구 정의를 자동으로 캐시 계층으로 밀어넣기 때문에 대부분의 prompt 가 `cache_creation_input_tokens` 로 흘러감.
- "print hello" 같은 최소 프롬프트에도 36,796 토큰이 cache 에 쓰임 — 이게 Claude CLI 의 **세션 baseline 크기**.
- v0.1 의 `input 5 / output 1230` 은 버그가 아니라 필드 해석 오류였음. script 호출의 실제 prompt load = 5 (fresh) + cache_creation + cache_read.

## usage 필드 전수 정리

| 필드 | 의미 | 비용 |
|---|---|---|
| `input_tokens` | 이번 호출에서 **새로 흘러들어온** non-cached 입력 | 정가 |
| `cache_creation_input_tokens` | 이번 호출에서 **캐시에 새로 쓴** 양 (다음 호출부터 재사용 가능) | 정가 + 25% |
| `cache_read_input_tokens` | 이전 캐시에서 읽은 양 | 정가 × 0.1 (90% 할인) |
| `output_tokens` | Claude 가 생성한 토큰 | 출력 정가 |
| `total_cost_usd` | CLI 가 계산한 실제 비용 | — |

**가장 정확한 단일 지표**: `total_cost_usd`. cache tier 가중치가 이미 반영됨.

## 결정

**옵션 (b) 채택** — cache 관련 필드를 별도 키로 별도 기록.

### 이유
- (a) "prompt byte 기반 대체 지표" 는 프롬프트 내용에 따른 토크나이저 오차가 큼. CLI 가 이미 정확한 값을 주는데 별도 추정할 이유 없음.
- (c) "현상 유지 + 주석" 은 rubric 의 토큰 효율 축을 설계할 수 없게 만듦. v0.2 E 에서 이 축이 필요함.
- (b) 는 스키마 변경 없이 (기존 `levelup_runs.tokens` 는 JSONB) 가능.

### 구현
- `src/claude.py:call()` — `tokens` dict 에 `cache_creation`, `cache_read` 키 추가. 4 필드 shape.
- `src/loop.py` — `empty_tokens` 4 필드. `run_script()` 의 stderr aggregation 도 4 필드 합산.
- `src/judge.py` — empty tokens return 도 4 필드.
- `src/db.py` — `AX_VERSION = "v0.2"` bump. `tokens` JSONB 는 추가 필드를 그대로 흡수.
- **dashboard 호환**: `dashboard/src/app/(dashboard)/tokens/page.tsx` 는 `t[part]?.input ?? 0` 식 옵셔널 체이닝으로 읽음. 새 필드는 무시되고 기존 `input`/`output` 키는 여전히 존재 — 깨지지 않음. dashboard 에 cache 필드 표시는 v0.3 에 정식 작업.

## rubric 의 토큰 효율 축 설계 (E 에 전달)

ax-implement rubric 의 "토큰 효율" 축은 다음 중 한 지표로 판정:

1. **권장: `total_cost_usd`** — 가장 정확. 같은 fixture 에 대해 iter 간 비교.
2. 보조: `output_tokens` 만 — "산출물 자체가 얼마나 길어졌나" 에 집중. 자연어 SKILL.md 가 장황하게 써서 output 이 커지는 경우 감지.
3. 비권장: `input + cache_creation + cache_read` 합 — 직관적이나 cache 할인율이 반영 안 됨.

**첫 run 은 기준선**. absolute threshold 는 v0.3 에서 정한다. v0.2 E 에서는 "점수 계산에 반영하되, 첫 run 의 값으로 정규화 후 이후 iter 가 그 이상이면 감점" 식의 상대 비교만.

## 향후 (v0.3+)

- **대시보드 Tokens 탭에 cache breakdown 차트 추가** — stage 별로 input / cache_creation / cache_read 분해.
- **세션 baseline 크기 추적** — Claude CLI 버전 / 도구 set 변경 시 baseline 이 튀는지 모니터.
- **캐시 warm-up 효과 측정** — 같은 세션 내 반복 호출에서 `cache_read` 증가가 실제 비용 감소로 이어지는지 확인.
