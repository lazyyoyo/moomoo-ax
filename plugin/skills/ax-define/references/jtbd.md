# JTBD (Jobs To Be Done)

team-ax `define` 단계의 첫 번째 프레임워크. **"왜·무엇"을 한 문장으로 묶는다.**

## 정의

JTBD = 사용자가 "이번 제품 버전을 통해 해결하려는 일(Job)" 한 줄.

- 기능 나열이 아니다. ("프로필 페이지 + 테마 모달 + 비밀번호 변경" → ✗)
- 사용자 관점의 결과다. ("내 프로필과 설정을 한 곳에서 관리한다" → ○)

## 레벨 구분 — JTBD / Story / Backlog

| 레벨 | 정의 | 변동성 | 예시 |
|---|---|---|---|
| **JTBD** | 제품 또는 제품 버전의 방향 | 거의 안 바뀜 | "내 프로필과 설정을 한 곳에서 관리한다" |
| **Story** | 기능 단위 — Story Map의 칸 | 버전 단위로 변동 | "프로필 보기", "테마 변경" |
| **Backlog** | 구현 가능한 작업 항목 | 평소에 쌓임 | "ISBN 스캔 추가", "다크 모드 토글" |

> team-ax에서는 **한 제품 버전(minor) = 하나의 JTBD**. JTBD가 복합적이면 Story Map에서 feature 단위로 분해한다. 버전을 쪼개지 않는다.

## "And 없는 한 문장" 테스트

JTBD 한 줄에 접속사 "and / 그리고 / 및 / + / ,"가 있으면 두 가지 관심사가 묶였다는 신호.

| 후보 | 판정 | 이유 |
|---|---|---|
| 사용자가 책을 추가한다 | ○ | 단일 관심사 |
| 내 프로필과 설정을 한 곳에서 관리한다 | ○ | "프로필/설정"은 같은 관심사(마이페이지)의 표현 |
| 사용자가 책을 추가하고 독서 진도를 기록한다 | ✗ | "추가"와 "기록"은 별개 Job — Story로 분해 |
| UX 패치 + 원서 검색 도입 | ✗ | 패치 모음과 신규 기능이 묶임 — Story로 분해하거나 JTBD 재정의 |
| 앱을 안정적으로 사용하고 데이터가 정확하게 표시된다 | ○ (패치 모음 예외) | 단일 관심사("안정성")의 두 측면 |

> 패치 모음 제품 버전(예: rubato 제품 v1.5.1)은 "안정성" 같은 추상 단어 하나로 묶일 수 있다. 이 경우 Story 없이 태스크 목록만 두는 것을 허용.

## define에서 JTBD가 하는 일

1. Phase A에서 **가장 먼저** JTBD 한 줄을 합의한다 (인터뷰 직후).
2. Story Map은 이 JTBD 아래에 펼쳐진다 — JTBD가 흐트러지면 Story Map도 흐트러진다.
3. SLC 체크의 **Complete** 축이 JTBD 충족 여부로 판정된다.
4. scope.md `§ JTBD` 섹션의 한 줄이 그대로 들어가고, 이후 어떤 단계도 이 줄을 바꾸지 않는다 (바뀌면 제품 버전을 새로 정의한 것).
