# doc 리뷰 체크리스트 (v0.1.2)

대상: 문서 수정이 의도대로 됐는지 검증. 1차 적용 = `versions/undefined/scope.md` (define Phase C 리뷰).

## 검증 입력

- **계획 측** — 대상 문서의 `§수정 계획` 표 (파일별 변경 타입 + 변경 요약 + 매핑 근거 + 대응 태스크)
- **실측 측** — `git diff HEAD` 또는 §수정 계획에 언급된 파일들의 현재 상태

> **평가 대상 한정 규칙**: 실측 측 diff에 §수정 계획에 등장하지 않는 파일이 포함되어 있을 수 있다 (사전 unstaged 잔재 등). 이런 파일은 **평가 대상에서 제외**한다 — PASS도 FAIL도 아닌, 판정하지 않음. 체크리스트 5종 항목은 §수정 계획에 명시된 파일에만 적용한다.

## 체크리스트 5종

### 1. §수정 계획의 각 항목이 실제 파일에 반영됐는가?

§수정 계획 표의 각 행에 대해:
- 갱신/마커 행 → 해당 spec 파일에 변경이 실재하고 마커가 부착됐는가?
- 신규 행 → 해당 파일이 실제로 생성됐고, §수정 로그에 체크가 있는가?
- 삭제 행 → 본 단계는 권장하지 않음 (deploy로 미룸). deprecated 마커만 부착됐는가?
- BACKLOG 정리 행 → BACKLOG.md inbox에서 해당 줄이 제거됐는가?

> 누락 1건이라도 발견되면 **FAIL**.

### 2. 새 파일 생성 시 "기존 파일 매핑 탐색" 근거가 §수정 계획에 있는가?

§수정 계획의 모든 신규 행에 매핑 근거 한 줄이 명시되어야 한다 — "기존 specs/ 전수 grep 후보 0개" 또는 "user.md(인증 도메인)와 분리 유지" 같은 명시적 근거.

> 근거 없음 → **FAIL** ("새 파일 남발" 가드 위반).

### 3. 파일명이 시점·변경 단위 접미사를 포함하지 않는가?

다음 접미사 모두 금지:

- `-fix`
- `-patch`
- `-enhance`
- `-redesign`
- `-v2`, `-v3`, ...
- `.old`, `.legacy`

§수정 계획의 신규/갱신 파일명, 그리고 `git status`의 신규 파일 모두 검사.

> 1개라도 발견 → **FAIL**.

### 4. spec 본문에 시간 축 기술이 없는가?

§수정 계획에 등장한 모든 spec 파일의 본문에 다음 표현이 있는지 검사:

- "v1.5에서 추가됨" / "v1.7부터 변경"
- "기존엔 X였는데 Y로 변경" (이력형 본문)
- 변경 이력 테이블 (`| 버전 | 변경 |` 형식)
- "Phase 1 / Phase 2" 식의 시점 누적 섹션

> 변경 이력은 git log + scope.md §수정 로그가 담당. spec 본문은 "오늘의 진실"만. 발견 시 **FAIL**.

### 5. 버그 수정·핫픽스가 신규 파일이 아닌 기존 spec 갱신으로 처리됐는가?

§수정 계획에 "버그 수정 / 핫픽스 / 패치" 성격의 행이 있다면:

- 신규 파일로 처리됐는가? → **FAIL** (장치 4 위반)
- 기존 spec in-place 갱신으로 처리됐는가? → PASS
- 시나리오 영향이 없어 spec을 건드리지 않는다면? → §수정 계획에 "spec 변경 없음 — CHANGELOG는 deploy 단계" 명시 필요. 명시 없으면 **FAIL**.

## 출력 포맷

```
{APPROVE | REQUEST_CHANGES: {한 줄 이유}}

## 항목별 근거
1. §수정 계획 반영도: PASS/FAIL — {근거: 누락 항목 또는 모두 반영}
2. 신규 파일 매핑 근거: PASS/FAIL — {근거: 매핑 누락 행 또는 모두 명시}
3. 파일명 접미사: PASS/FAIL — {근거: 위반 파일명 또는 위반 없음}
4. 시간 축 본문: PASS/FAIL — {근거: 위반 파일:라인 또는 위반 없음}
5. spec vs CHANGELOG: PASS/FAIL — {근거: 위반 행 또는 분리 유지}

## 재작업 노트 (REQUEST_CHANGES일 때만)
- {수정 지침 1}
- {수정 지침 2}
```

## 판정 규칙

- 5개 항목 모두 PASS → `APPROVE`
- 1개라도 FAIL → `REQUEST_CHANGES: {가장 심각한 항목 한 줄 요약}`
